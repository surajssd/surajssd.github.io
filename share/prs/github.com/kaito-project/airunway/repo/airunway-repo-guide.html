? What problem does it solve?

Running an LLM on Kubernetes usually means learning a specific stack — KAITO, NVIDIA Dynamo, KubeRay, llm-d — each with its own CRDs and YAML. AI Runway puts one abstraction on top of all of them: you describe what model you want, and it figures out how and where to run it.

Unified API

A single ModelDeployment CRD works across every provider. Write it once; the controller translates it into provider-specific resources.

Auto-selection

You can omit the engine and provider. The controller uses CEL rules + capability matching to pick the best fit for your hardware and model.

Click-to-deploy UI

Browse HuggingFace, check GPU fit & cost, deploy — all from a web UI. The UI is fully optional & swappable.

{ } The heart: ModelDeployment CRD

This is the one object you really need to understand. The simplest valid deployment is just a model ID — everything else is auto-filled.

Minimal — let it decide everything

apiVersion: airunway.ai/v1alpha1
kind: ModelDeployment
metadata:
  name: my-model
spec:
  model:
    id: "Qwen/Qwen3-0.6B"

The controller picks the engine, the provider, the serving mode, and creates the provider-native resource.

Fuller — explicit control

spec:
  model:
    id: "Qwen/Qwen3-0.6B"
    source: huggingface
  engine:
    type: vllm        # or sglang/trtllm/llamacpp
  provider:
    name: kaito       # optional explicit pick
  serving:
    mode: aggregated  # or disaggregated
  resources:
    gpu: { count: 1 }
  gateway:
    enabled: true

The second CRD: InferenceProviderConfig (cluster-scoped)

Each provider registers itself with a config that declares what it can do. The controller reads these to make selection decisions.

capabilities

Which engines, serving modes, GPU/CPU support, gateway features it offers.

selectionRules

CEL expressions + priority that decide if this provider matches a given ModelDeployment.

status.ready

Health + heartbeat + upstream CRD version/schema hash for drift detection.

↻ What happens when you deploy

From clicking "Deploy" in the UI to a running inference pod serving an OpenAI-compatible API.

User picks a model in the UI

Browse HuggingFace catalog → backend checks GPU fit & cost → submits a ModelDeployment via the REST API.

Webhook validates the spec

The validating webhook checks compatibility (engine ↔ provider ↔ hardware) and rejects invalid or unsafe specs before they're stored.

Controller selects a provider

Runs the selection algorithm: matches required capabilities, evaluates each provider's CEL selectionRules by priority, picks a winner.

Provider controller takes over

The core controller delegates; the out-of-tree provider controller translates the spec into its native CRD (KAITO Workspace, Dynamo Graph, RayService…).

Pods spin up

The engine (vLLM / SGLang / TRT-LLM / llama.cpp) starts on GPU/CPU nodes and pulls model weights from HuggingFace.

Status flows back & gateway wires up

Status aggregates to Running; if Gateway API CRDs exist, an InferencePool + HTTPRoute expose the model on a unified endpoint.

⚙ The four providers

Each provider is its own Go module under providers/ — an out-of-tree controller (a "shim") that bridges AI Runway's CRD to an upstream project. Sizes below are Go LOC.

🟢 NVIDIA Dynamo ~6.2k LOC

GPU-accelerated inference with aggregated or disaggregated serving. Translates to a Dynamo Graph. The largest provider shim.

🩷 KAITO ~5.9k LOC

vLLM (GPU) and llama.cpp (CPU/GPU). Maps to a KAITO Workspace. The only provider with CPU-only support.

🔵 KubeRay ~3.4k LOC

Ray-based distributed inference. Produces a RayService for multi-node serving.

🟠 llm-d ~2.9k LOC

vLLM (GPU) with aggregated or disaggregated serving. Lighter-weight vLLM-native path.

Go Controller internals

The Go operator under controller/ is the brain. Built with Kubebuilder, it follows the standard reconcile loop pattern.

🎯 modeldeployment_controller.go

The main reconciler. Validates spec, runs the selection algorithm, delegates to providers, aggregates status.

🛡 webhook/v1alpha1/

Validating webhook — compatibility checks, security validation, cached reader. Blocks bad specs at admission time.

🌐 gateway_reconciler.go + gateway/detection.go

Auto-detects Gateway API CRDs; if present, creates InferencePool + HTTPRoute per deployment.

🔀 migration.go

Handles API/version migrations of existing resources across controller upgrades.

📊 metrics/metrics.go

Prometheus metrics (recently added — see ServiceMonitor & o11y commits).

🧩 validation/compatibility.go

Single source of truth for which engine works with which provider & hardware.

Test coverage is strong — nearly every *.go file ships a *_test.go sibling (suite_test, run_selection_algorithm_test, security_validation_test, etc.).

TS Web UI layer

The shared contract

📦 @airunway/shared

TypeScript types + a typed API client used by both the React UI and the Headlamp plugin. Includes model, costs, autoscaler, huggingface, installation, settings types — and generated versions.generated.ts.

⤞ Gateway API integration

An optional, auto-detected feature: serve all your models through a single endpoint using body-based routing on the model field.

1 · Detect

At startup the controller checks for Gateway API Inference Extension CRDs. Absent → feature silently disabled.

2 · Wire

Present → for each ModelDeployment it creates an InferencePool + HTTPRoute.

3 · Route

One gateway endpoint; the model field in the request body picks the backend. Demo in demos/gateway-bbr.

$ Developer cheat-sheet

Top-level Makefile drives everything. Controller and UI have separate workflows.

Controller (Go)

# build / test / run the operator
make controller-build
make controller-test
make controller-run

# after editing *_types.go
cd controller && make manifests generate

# install CRDs / deploy to cluster
make controller-install
make controller-deploy

Web UI (TypeScript)

# install + dev servers (FE + BE)
bun install
bun run dev
bun run test

# build single binary → dist/
make compile
make compile-all   # all platforms

Fastest way to try it

# 1. Install CRDs + controller
kubectl apply -f https://raw.githubusercontent.com/kaito-project/airunway/main/deploy/controller.yaml
# 2. Install the dashboard UI (optional)
kubectl apply -f https://raw.githubusercontent.com/kaito-project/airunway/main/deploy/dashboard.yaml
kubectl port-forward -n airunway-system svc/airunway 3001:80
# 3. Apply a provider shim (e.g. KAITO), then deploy a model from the UI
