The repo in three sentences (for a first-time reader)

AI Runway is a Kubernetes operator that lets you say "deploy this model" once and have several different inference stacks actually run it. Users create a ModelDeployment custom resource; the controller looks at registered InferenceProviderConfigs (one per provider — kaito, dynamo, kuberay, llmd) and picks the right one based on engine type (vLLM, SGLang, TRT-LLM, llama.cpp), GPU/CPU requirements, and serving mode (aggregated vs disaggregated). A validating webhook rejects bad specs up front, and a gateway reconciler optionally wires the deployment into the Gateway API Inference Extension (GAIE).

Why this PR matters

Before this PR, a provider declared things like cpuSupport: true, servingModes: [aggregated], and gateway: {...} once, and that applied to every engine the provider supported. That is wrong in practice:

  • KAITO supports both vLLM (GPU only) and llama.cpp (also CPU). A flat cpuSupport can't say that.
  • Dynamo's TRT-LLM engine only does aggregated mode; vLLM and SGLang do both.
  • Gateway capabilities differ per engine: KAITO's llama.cpp ignores servedName; vLLM does not.

Migration flow at controller startup

This runs once the manager wins leader election and before Reconcile is allowed to fire.

Why the migration is non-trivial. If you just deserialize a legacy CR into the new Go struct, the flat fields silently drop because they have no place to land. The unstructured-client approach keeps a copy of the raw JSON, hoists fields into each engine, then re-issues an Update - surviving HA controller races via retry.RetryOnConflict with a fresh Get.

Findings & risks

Self-rated severity. Click a row to expand.

CRIT legacyFlatKeys is a forward-compat tripwire

The migration unconditionally deletes top-level keys named servingModes / gpuSupport / cpuSupport / requiresCRD / gateway from spec.capabilities. If a future change reintroduces a top-level field whose JSON tag overlaps, that field will be silently stripped on every controller restart.

Mitigation already in PR: a doc comment on ProviderCapabilities and on legacyFlatKeys warns explicitly. Worth pairing with a unit test that fails if the names ever collide.

MED Breaking CRD change for off-cluster consumers

Anything that constructs InferenceProviderConfig outside the cluster (Helm chart values, CI fixtures, Headlamp plugin tests, third-party providers vendoring the API package) must regenerate. The in-cluster migration only covers stored objects, not source.

MED Webhook now fails closed on RBAC errors

Previously, a webhook RBAC failure when reading InferenceProviderConfig would surface as a warning. After 7d538b5 it denies admission. This is correct (safer default) but is observable: a misconfigured RBAC on upgrade will look like "ModelDeployments suddenly stop accepting" rather than "validation produced a warning".

OK Reconciler no longer mutates md.Spec.Engine.Type

Commit 155034f stops the reconciler from writing back the auto-selected engine onto user-owned spec. The resolved engine is threaded through reconcile as a local instead. This brings the reconciler closer to the kubebuilder "spec is user-owned, status is mine" contract.

OK EngineSelected=True now deferred until validation passes

Commit f9b570d: previously a ModelDeployment that failed compatibility checks could still flip its EngineSelected condition to True, misleading observers. Now it stays unset until the full chain is happy.
