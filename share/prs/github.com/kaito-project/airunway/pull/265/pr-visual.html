This PR adds a sixth inference provider, Direct vLLM, to AI Runway. Unlike the existing providers that render a ModelDeployment into an upstream operator CR (KAITO Workspace, Dynamo, RayService, llm-d), the vLLM provider renders native apps/v1 Deployment + Service objects directly — no operator dependency, aimed at “newest model support and configurable launch images.” It ships as a standalone controller under providers/vllm/ (controller, transformer, image resolver, status translator, RBAC, Dockerfile, ~3,500 lines of tests).

Two cross-cutting capabilities ride along. First, new CRD surface: spec.engine.image (preferred image override), spec.engine.extraArgs (raw passthrough flags), and a rich status.image block that records the registry-resolved digest plus provenance. Second, a backend vLLM recipe pipeline (recipes.vllm.ai) that the Web UI uses to pre-fill deployment fields, hardened against SSRF / path-traversal / oversized-payload abuse.

The commit shape is striking: only 2 of 20 commits are feat; the other 18 are fix/build/ci/docs, and at least eight are explicitly “address review feedback.” The feature landed rough and was hardened in-branch — reassuring for test depth, but it means the diff mixes the original feature with many small correctness patches across other providers and the shared validation layer.

Scrutinize first: the reconcile loop now makes a synchronous remote registry round-trip (setImageResolutionStatus → RemoteImageResolver.Resolve) to pin the default cu130-nightly tag to a digest, and for the provider-default image a resolution failure is a hard reconcile error. It is cached and 30s-bounded, but a registry outage on a cold cache blocks every default-image deployment from progressing. The second thing to check is the provider-switch orphan guard (issue #325): switching spec.provider.name after selection is rejected rather than handled, and deselection never cleans up the old provider’s resources.

Glossary & Concept Primer

Vocabulary an unfamiliar reviewer needs before the rest of the dashboard makes sense.

ModelDeployment (MD)
The unified CRD users apply. A provider turns it into concrete cluster resources. This PR teaches the vLLM provider to turn it into a Deployment + Service.
Provider
A controller that reconciles a ModelDeployment into some backend’s resources. Existing: kaito, dynamo, kuberay, llmd. New: vllm (“Direct vLLM”).
Direct vLLM
The new provider. “Direct” = it emits stock Kubernetes Deployment+Service running the vllm/vllm-openai OpenAI-compatible server image, with no intermediate operator.
Transformer
The pure function inside a provider that maps an MD spec to unstructured K8s objects (providers/vllm/transformer.go). Builds the container, args, probes, volumes.
engine.args vs engine.extraArgs
engine.args is a structured map[string]string of vLLM flags. engine.extraArgs (new) is a raw []string appended verbatim. A flag appearing in both is now rejected at admission.
Recipe
A JSON document on recipes.vllm.ai describing a known-good launch command for a model. The backend fetches and “resolves” it into deployment fields (GPU count, engine args, image).
Resolve (recipe)
Backend logic (vllmRecipeResolver.ts) that parses a recipe’s vllm serve … command line into structured engineArgs, engineExtraArgs, GPU count, image ref, and provenance annotations.
Image resolution / digest pinning
The controller queries the registry to turn a moving tag (cu130-nightly) into an immutable repo@sha256:… digest, recorded in status.image and baked into generated pods.
Provenance annotations
airunway.ai/recipe.* + airunway.ai/generated-by annotations stamped onto a manifest so you can tell a recipe-generated deployment from a hand-written one.
Aggregated vs Disaggregated serving
Aggregated = one pod does prefill+decode. Disaggregated = separate prefill/decode Deployments sharing a KV cache. Direct vLLM advertises aggregated only; disaggregated rendering exists internally but is rejected at reconcile.
Validating webhook vs reconcile backstop
Two enforcement points. The webhook rejects bad specs synchronously at kubectl apply time. The reconcile backstop re-checks the same rule for objects that bypass admission (tests, upgrades, disabled webhooks).
versions.env / ldflags injection
Single source of truth for pinned versions. VLLM_VERSION=cu130-nightly is injected into the provider binary via -ldflags -X …VLLMVersion=; a Go literal fallback must match (enforced by make verify-versions).

External dependency / availability WARN Reconcile makes a synchronous registry round-trip to digest-pin the image. For the provider default, resolution failure is a hard reconcile error (controller.go). Cached + 30s-bounded, but a registry outage on a cold cache blocks all default-image deployments. Verify the cache (reuseResolvedImageStatus) survives controller restarts — status is persisted, so it should, but worth confirming.
Lifecycle / migration safety WARN Provider switch after selection is rejected, not handled (#325). Deselection never GCs the old provider’s resources/finalizer. Acceptable as an interim guard, but a user’s only recovery is delete + recreate — confirm that’s surfaced in user-facing UI text, not just code comments.
API contract / backward compatibility WARN ModelDeploymentSpec.env changed shape in shared TS types from Record<string,string> to EnvVar[]. The in-repo conversion is handled, but any external consumer of the manifest (notably the Headlamp plugin, not inspected here) that read spec.env as a map will break. The Go CRD already used a structured env, so this is a TS-side alignment — still a breaking type change.
Security (input handling, SSRF, secrets) INFO Strong, and clearly the focus of several commits: HTTPS-only + same-origin + path-prefix checks on recipe refs, strict <org>/<model> validation + encodeURIComponent, 5 MiB streamed size bound, 10s timeout, single-segment route. HF token via secretKeyRef with duplicate-drop. Residual: a recipe can still inject an arbitrary image ref resolved to source custom — the trust boundary is “you trust recipes.vllm.ai.” Documented.
Error handling / failure modes INFO Generally careful: finalizer timeout is enforced even when the owned Deployment is stuck Terminating (Delete returns nil); syncStatus failure no longer downgrades a Running deployment back to Deploying; typed recipe errors map to 400/502/504. These are exactly the subtle bugs the review tail fixed.
Performance on hot paths INFO Recipe index + per-model payloads cached (TTL + LRU 256). Image digest reused once resolved. The periodic RequeueAfter status sync is standard. No obvious hot-loop regression; the registry call is the main added latency (see WARN above).
Concurrency / races INFO Heartbeat goroutine is a standard ticker with ctx cancellation. The JS recipe cache is single-threaded (no lock needed) but allows a cache-stampede of concurrent identical fetches — harmless, just redundant upstream calls. No shared-state races spotted.
Observability INFO Good: 6 new conditions, a full status.image block with human messages, recipe provenance annotations. Gap: ImageVerified is permanently Unknown (no attestation) — correct to surface, but downstream tooling must not treat it as a security signal.
Test coverage of new behavior OK Strong. ~95 Go test cases (transformer 1,421 lines, controller 497, image-status 354) and ~26 TS cases covering cache eviction, streaming bound, arg dedup, HF_TOKEN dedup, finalizer timeout, GPU-per-pod derivation, the --model guard, SSRF rejections, and 301 trailing-slash behavior. CI matrix updated to run vllm checks.
Dead / unadvertised code path INFO transformDisaggregated is fully implemented and tested, but validateCompatibility rejects disaggregated before Transform runs. The rejection precedes Transform in Reconcile (confirmed in read), so the path stays unreachable in production as intended — just note it as carried-but-dormant code.
