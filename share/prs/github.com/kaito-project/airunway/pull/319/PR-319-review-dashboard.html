fix(backend): honour kubeconfig CA under Bun's native fetch

PR #319 on GitHub branch suraj/fix-frontend-authn-issues → main @surajssd 7 files, +127 / -20 1 commit Bug fix · Refactor

Executive Summary

The Web UI backend runs on Bun, not Node.js. The @kubernetes/client-node SDK hands the kubeconfig's TLS material (the cluster CA, plus any client cert/key) to its HTTP layer as a Node.js https.Agent. Bun's native fetch silently ignores that agent — it only honours TLS material passed via its own per-request tls option. So every request to a cluster with a private CA (e.g. AKS) failed with UNABLE_TO_VERIFY_LEAF_SIGNATURE.

This PR adds two things to backend/src/lib/kubeconfig.ts: BunTlsHttpLibrary, a subclass of the SDK's IsomorphicFetchHttpLibrary that overrides send() to call fetch directly while re-injecting the CA/cert/key as tls; and a makeApiClient(kc, ApiClass) helper that re-creates the SDK's own client wiring but swaps in that HTTP library. All six backend services (auth, autoscaler, config, kubernetes, registry, secrets) are switched from kc.makeApiClient(...) to the new helper. The approach copies a pattern already proven in proxyServiceRequest.

What a reviewer should scrutinise most: TLS verification correctness. The single highest-stakes line is the rejectUnauthorized = false branch — get its condition wrong and you silently disable certificate verification for every cluster call. Tied for attention: this is a security-sensitive auth fix with zero automated tests and the PR's own checklist shows bun run test / bun run lint were not run.

Glossary & Concept Primer

Vocabulary an unfamiliar reviewer needs before the rest of the dashboard makes sense.

Bun
A fast JavaScript/TypeScript runtime (alternative to Node.js) that the Web UI backend runs on. Its built-in fetch is WHATWG-native and does not understand Node's https.Agent; TLS config goes through a non-standard tls option on the request.
kubeconfig / KubeConfig
The credentials file (usually ~/.kube/config) describing how to reach a cluster: API server URL, the cluster CA certificate, and the user's auth (token or client cert/key). Represented in code by the SDK's k8s.KubeConfig object.
CA (Certificate Authority)
The root cert used to verify the API server's TLS certificate. Managed clusters like AKS use a private CA not in the system trust store, so the client must be told about it explicitly.
UNABLE_TO_VERIFY_LEAF_SIGNATURE
The TLS error thrown when the client can't chain the server's cert back to a trusted CA — i.e. the exact symptom this PR fixes.
@kubernetes/client-node (the SDK)
The official JS Kubernetes client. kc.makeApiClient(ApiClass) normally builds a typed API client (e.g. CoreV1Api) wired to the kubeconfig.
IsomorphicFetchHttpLibrary
The SDK's default HTTP transport. Its send() uses node-fetch and passes TLS via a Node https.Agent — the part Bun ignores. BunTlsHttpLibrary subclasses it and overrides send().
applyToHTTPSOptions()
SDK method that extracts ca/cert/key/rejectUnauthorized from the kubeconfig into a plain options object — used here to translate into Bun's tls option.
rejectUnauthorized / skipTLSVerify
skipTLSVerify is the kubeconfig flag; rejectUnauthorized: false is its TLS-layer equivalent. When set, certificate verification is disabled — intentional only for dev clusters.
authMethods / Bearer token
The SDK applies auth headers (e.g. Authorization: Bearer <token>) to the request before send() runs, so the override only needs to re-inject TLS, not auth.
proxyServiceRequest
An existing method in kubernetes.ts that already does the same manual TLS-into-fetch translation for in-cluster service proxying — the proven pattern this PR generalises.
AUTH_ENABLED cert-stripping
When auth is on, loadKubeConfig() deletes the admin client cert/key from the kubeconfig so that Bun's process-wide shared TLS session can't accidentally authenticate user requests as admin.

Annotated Diff

All substantive hunks shown. The five service files (auth, autoscaler, config, registry, secrets) share one mechanical edit pattern, shown once below; kubernetes.ts has the same plus per-user-token call sites. The real logic is all in kubeconfig.ts.

backend/src/lib/kubeconfig.ts new: BunTlsHttpLibrary.send() 
export class BunTlsHttpLibrary extends k8s.IsomorphicFetchHttpLibrary {
+  constructor(private readonly kc: k8s.KubeConfig) { super(); }
+  send(request: k8s.RequestContext): k8s.Observable<k8s.ResponseContext> {
+    const responsePromise = (async () => {
+      const httpsOptions = {};
+      await this.kc.applyToHTTPSOptions(httpsOptions as any);
+      const tls: Record<string, unknown> = {};
+      if (httpsOptions.ca)   tls.ca = httpsOptions.ca;
+      if (httpsOptions.cert) tls.cert = httpsOptions.cert;
+      if (httpsOptions.key)  tls.key = httpsOptions.key;
+      if (this.kc.getCurrentCluster()?.skipTLSVerify || httpsOptions.rejectUnauthorized === false)
+        tls.rejectUnauthorized = false;
+      const fetchOptions = { method, body, headers: request.getHeaders(), signal: request.getSignal() };
+      if (Object.keys(tls).length > 0) fetchOptions.tls = tls;
+      const response = await fetch(request.getUrl(), fetchOptions);
+      return new k8s.ResponseContext(response.status, headers, { text, binary });
+    })();
+    return new k8s.Observable(responsePromise);
+  }
}
Why: The whole fix lives here. By subclassing the SDK's HTTP library and overriding send(), every typed API client can transparently get Bun-compatible TLS. The key insight is re-reading request.getHeaders() (auth already applied) and translating applyToHTTPSOptions output into Bun's tls option. Scrutinise the rejectUnauthorized = false condition — it is the only line that can silently disable verification.
backend/src/lib/kubeconfig.ts new: makeApiClient() helper 
+export function makeApiClient<T extends k8s.ApiType>(
+  kc: k8s.KubeConfig, apiClientType: k8s.ApiConstructor<T>): T {
+  const cluster = kc.getCurrentCluster();
+  if (!cluster) throw new Error('No active cluster!');
+  const configuration = k8s.createConfiguration({
+    baseServer: new k8s.ServerConfiguration(cluster.server, {}),
+    authMethods: { default: kc },
+    httpApi: new BunTlsHttpLibrary(kc),
+  });
+  return new apiClientType(configuration);
+}
Why: A drop-in replacement for kc.makeApiClient() that reproduces the SDK's own wiring but injects BunTlsHttpLibrary as the transport. This reproduces SDK internals (createConfiguration, ServerConfiguration) — a maintainability tradeoff: it works today but couples the code to non-public SDK shape that could drift on upgrade.

Risk Assessment

Severity: CRIT WARN INFO OK

AxisSeverityFinding
Security — TLS verification OK Verified correct. rejectUnauthorized = false is set only when skipTLSVerify is true or rejectUnauthorized === false is explicit. Confirmed against SDK config.js:139-142 ("only set if explicitly configured"). No silent bypass.
API contract — tls-server-name WARN The SDK encodes cluster.tlsServerName into the agent's servername (config.js:147,172-173), which this PR discards. Clusters reached by IP or with a cert-name override lose hostname verification override. Map cluster.tlsServerName → tls.serverName. (Also absent in proxyServiceRequest.)
API contract — proxy-url WARN kubeconfig proxy-url is handled by the SDK's createAgent (config.js:434-449), which this PR bypasses. Users behind a kubeconfig-configured HTTP/SOCKS proxy regress. Map to Bun's fetch({ proxy }) or document as unsupported.
Test coverage WARN Zero automated tests for security-sensitive TLS/auth code. PR checklist confirms bun run test and bun run lint were not run. At minimum: a test that the default path does not set rejectUnauthorized:false (regression guard against silent disable).
Maintainability — duplication WARN TLS-extraction logic is now duplicated between BunTlsHttpLibrary.send() and proxyServiceRequest (kubernetes.ts:2170-2194). They can silently drift (e.g. one gains serverName, the other doesn't). Extract a shared kubeConfigToBunTls(kc) helper. (Flagged by 4 of 4 completed reviewers.)
Maintainability — SDK-internals coupling INFO makeApiClient hand-reproduces createConfiguration + ServerConfiguration + subclasses IsomorphicFetchHttpLibrary. Works on @kubernetes/client-node@1.4.0; could break on a major SDK bump. Add a version-pin note and a smoke test.
Correctness — per-request re-run of applyOptions WARN Verified. send() calls applyToHTTPSOptions → applyOptions, which re-runs both applyHTTPSOptions and applyAuthorizationHeader (config.js:517-519) on every request — even though the generated client already applied auth before send(). Effects: exec/OIDC auth plugins (kubelogin, aws eks get-token) run twice per call, and cert files re-read from disk (bufferFromFileOrString, config.js:529-531). Cache the TLS material once per client. (Root cause behind Claude's "double credential resolution", Agency's "disk I/O", Codex's "auth re-run".)
Robustness — request body INFO Body is cast to BodyInit; the SDK's Node form-data multipart type wouldn't serialise under Bun's fetch. None of the 6 API classes used post multipart today, so safe — worth a guard/comment for future callers.
Robustness — error / non-2xx handling INFO Matches upstream: non-2xx responses pass through as ResponseContext (typed APIs map to errors). fetch rejections (DNS/TLS/abort) propagate via the Observable promise. Behaviour preserved.
Performance — hot path INFO Per-user-token calls allocate a fresh KubeConfig + BunTlsHttpLibrary per request. Functionally fine; a token-keyed cache is a possible optimisation if GC pressure appears. binary() buffers the whole body (fine for K8s JSON, unsuitable for streaming — but backend does no watches).
Observability INFO No new logging in send(). TLS failures surface as raw fetch errors. Acceptable; a debug log of which TLS material was applied (without secrets) would aid future diagnosis.
Auth-header preservation OK Verified: SDK applies authMethods to the RequestContext before send(); override re-reads request.getHeaders(), so Bearer tokens survive. Respects AUTH_ENABLED cert-stripping (reads the already-stripped user).
Concurrency / races INFO Not applicable — clients are constructed once (or per request) with no shared mutable state introduced. Note Bun's documented process-wide TLS session sharing is handled separately in loadKubeConfig.

Assumptions & Unknowns

Everything I'm guessing, couldn't verify, or would need to ask the author about. All hedging lives here.

  • Verified against SDK source. Findings citing config.js / http.js line numbers were checked against the actually-installed @kubernetes/client-node@1.4.0 in node_modules. Confirmed: getSignal() does exist (http.js:107), so the "runtime crash" one reviewer reported is a false positive; tlsServerName/SNI and proxyUrl drops are real; applyOptions re-running auth per request is real.
  • Runtime behaviour not executed. I did not run bun run test or stand up a live cluster. The "works against AKS" claim rests on the PR author's manual testing notes (200s on /api/cluster/status etc.), not on a reproduction here.
  • SNI / proxy severity is contextual. The tls-server-name and proxy-url gaps only bite clusters that actually set those kubeconfig fields. Standard AKS/EKS kubeconfigs usually don't, which is why the author's manual test passed. Would need to confirm the target deployment environments never use a proxy-url or cert-name override to downgrade these from WARN.
  • Bun connection-pool cross-contamination (one reviewer, MEDIUM) is unverified. The claim that Bun may reuse a pooled insecure connection for a secure client to the same origin is plausible but I did not confirm Bun's pooling semantics. If true it would be serious; worth a targeted test before relying on it either way.
  • Multipart bodies. I confirmed none of the 6 API classes used (CoreV1Api, AppsV1Api, CustomObjectsApi, AuthenticationV1Api, AuthorizationV1Api, ApiextensionsV1Api) post form-data multipart today, so the BodyInit cast is safe — but this is a property of current call sites, not an invariant.
  • Was the duplication with proxyServiceRequest a deliberate choice? The PR comment explicitly says it "mirrors" that pattern. Unclear whether the author intends a follow-up to DRY them, or considers two copies acceptable. All four substantive reviewers flagged it.
  • This dashboard's diagrams are my reconstruction of the control flow from reading the diff and SDK, not generated from a trace. The sequence diagram's ordering (auth-before-send) is verified from CoreV1Api.js + config.js, but treat box-and-arrow flows as explanatory, not authoritative.
